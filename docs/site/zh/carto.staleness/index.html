Staleness Lifecycle

这是一个概念节点：它端到端地追踪一个 vault 的新鲜度生命周期，横跨 CLI 与哈希核心，而不是记录某个文件。

职责

陈旧（staleness）只回答一个问题：哪些页面可能已经与它所描述的源代码不再一致？ carto 通过对每个被引用的 source 计算哈希、并将当前哈希与 ir.json 中存储的哈希相比对，来确定性地追踪这一点。关键在于，陈旧是建议性的——它报告漂移，但从不阻断任何事，也从不改动你的正文。

“陈旧”是相对什么衡量的

ir.json 里的每个 source 都携带一个 sha256:<hex> 形式的 hash。那个存储的哈希，是你上一次运行 carto hash 时对该文件拍下的快照。当一个节点引用的某个 source 如今哈希出的值与存储值不同，这个节点就被视为陈旧。因此陈旧是逐 source 的：它是单个 anchor:file 对的属性，一个节点被标记，只是因为它的某个 source 发生了漂移。

数据流——从 pending 到一条构建提示

按顺序把生命周期走一遍：

  1. 作者写下一个 source。 当你给一个节点添加 source 时，把它的 hash 设为占位符 sha256:pending。你是在声明“这个文件支撑这个节点”，却还不知道它的哈希。
  2. carto hash 记下真相。 该命令把每个 source 对照其锚点解析，从磁盘读取文件，计算 sha256:<hex>，再把真实值写回 ir.json。pending 占位符变成具体哈希。读文件与哈希这一步见 IR & Source I/O，哈希函数本身见 Hashing Core。
  3. 一个 source 被编辑。 之后，有人改动了其中一个文件。它的内容现在哈希出一个新值，但 ir.json 仍持有旧值——两者不一致。
  4. carto diff 找出漂移。 diff 重新计算当前哈希，列出每个当前哈希与存储哈希不同（或如今缺失）的 source，以 anchor:file 形式打印。这是最原始的文件级信号。
  5. carto stale 把漂移映射到页面。 stale 拿到发生变化的 source，报告引用它们的节点 id，并各自附上出问题的文件。这是报告，而非闸门——它告诉你哪些页面可能过时；是否以及何时重写，完全是你的决定。
  6. carto build 浮现提示。 构建期间会跑同一套陈旧检查，陈旧的节点 id 被传给站点模板，模板给这些页面标上提示。页面照样构建、照样发布——提示是一记提醒，而非错误。见 Site Build Driver。

diff、stale 与构建期检查都依赖 Hashing Core 中同一对纯函数：一个算出发生变化的 source 集合，另一个把变化的 source 映射到引用它们的节点。

建议性，从不作闸门

这条生命周期里没有任何东西会替你重写内容或让构建失败。stale 是一份你去读的报告，即便页面陈旧，build 依然产出一个完整的站点。刷新一个页面——连同它的全部语言一起重写并重跑 carto hash——是你做的决定，而非工具强加的。像本页这样的概念节点不引用任何 source，因此根本不会出现在陈旧报告里。

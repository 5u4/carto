职责

src/io.ts 是 carto 中唯一的文件系统边界。每一次读取 ir.json、每一次写回 ir.json、以及每一次读取被引用的源文件，都要经过这个模块。把所有磁盘访问集中在此，意味着流水线的其余部分——Hashing Core、CLI Commands 和 Site Build Driver——只需操作内存中的纯数据，永远不必直接接触 node:fs。

关键函数

  • readIr(irPath)——以 UTF-8 读取文件，JSON.parse 之后再交给 IrSchema.parse 校验（见 IR Schema）。校验并非可选：格式错误或结构非法的 IR 会在此处抛出，因此坏的 IR 会当场高声失败，而不会污染后续阶段。
  • writeIr(irPath, ir)——用 JSON.stringify(ir, null, 2) 序列化，并在写入 UTF-8 前追加一个换行符。两空格的美化缩进加上末尾换行，使 ir.json 对 diff 友好，且在多次重写之间保持稳定。
  • currentHashes(ir, contentDir)——解析并哈希每一个被引用的源文件，返回一个以 sourceKey 为键的 Map<string, string>。它是核心主力函数；本页其余部分将逐步走读它。

数据流

currentHashes 在有界并行下计算磁盘上每个源文件的实时哈希：

  1. 收集。 collectSources(ir)（来自 Hashing Core）遍历每个节点的 sources，返回去重后的 SourceRef[]——被两个节点同时引用的相同 anchor:file 只哈希一次。
  2. 展开并发。 一个共享游标 next 对源文件列表进行索引。模块启动 Math.min(HASH_CONCURRENCY, sources.length) 个 worker promise，其中 HASH_CONCURRENCY = 16。每个 worker 循环，通过 next++ 认领下一个源文件，直到列表耗尽——这是一个有界池，同一时刻打开的文件永不超过 16 个。
  3. 解析路径。 对每个源文件，worker 先解析锚点目录——resolve(contentDir, ir.vault.anchors[source.anchor])——再将 source.file 相对该锚点目录解析。因此锚点是相对内容目录解释的，而文件相对其锚点解释。
  4. 读取 + 哈希。 worker 用 readFile 读取字节并交给 hashContent，将结果存入以 sourceKey(source.anchor, source.file) 为键的位置。
  5. 跳过还是暴露。 以 ENOENT 失败的读取会被 continue 吞掉：缺失的文件不产生任何 map 条目，因此下游代码将其读作“不存在”，而非错误。任何其他错误——例如源路径指向目录时的 EISDIR——都会被重新抛出并从池中传播出去。
  6. 汇合。 await Promise.all(pool) 等待所有 worker，返回填充好的 Map。
const anchorAbs = resolve(contentDir, ir.vault.anchors[source.anchor]!);
const fileAbs = resolve(anchorAbs, source.file);
try {
  hashes.set(sourceKey(source.anchor, source.file), hashContent(await readFile(fileAbs)));
} catch (error) {
  if ((error as NodeJS.ErrnoException).code === "ENOENT") {
    continue;
  }
  throw error;
}

  • 将所有哈希与源枚举工作委托给 Hashing Core（collectSources、hashContent、sourceKey）。
  • 是 CLI Commands 每个子命令背后的读写引擎，也是 Site Build Driver 中陈旧度计算的基础。

Responsibility

src/build.ts is the bridge between carto’s hashing pipeline and the Astro Site Template. Its job is two-fold: compute which nodes are stale, then shell out to Astro to render the static site — handing the stale set to the template through the environment. It exposes one function, build, which the build subcommand of the CLI Commands calls.

Key types and functions

  • BuildOptions — the input contract:

    export interface BuildOptions {
  irPath: string;
  contentDir: string;
  outDir: string;
}

  • build(options) — reads the IR, derives the stale node ids, spawns Astro, and resolves when the child process exits cleanly.

Data flow

  1. Read + validate. readIr(options.irPath) loads and schema-checks the IR through the IR & Source I/O boundary.

  2. Diff. diffHashes(ir, await currentHashes(ir, options.contentDir)) compares each node’s stored hash against the live on-disk hash and returns the list of changed sources.

  3. Reduce to stale ids. The changes are mapped to { anchor, file }, passed to staleNodes (from the Hashing Core), and each stale node’s id is collected into staleIds.

  4. Locate the template and the Astro bin. The template directory is resolved relative to the compiled module — resolve(dirname(fileURLToPath(import.meta.url)), "..", "astro") — so it points at the sibling astro/ template dir regardless of cwd. The Astro executable is found via require.resolve("astro/package.json"), then joined to bin/astro.mjs.

  5. Spawn. spawn(process.execPath, [astroBin, "build", "--root", templateDir], …) launches astro build as a child of the current Node binary with stdio: "inherit", so Astro’s output streams straight to the user’s terminal. Three environment variables carry carto’s state into the build:

    • CARTO_CONTENT_DIR — the absolute content directory (resolve(options.contentDir)).
    • CARTO_OUT_DIR — the absolute output directory (resolve(options.outDir)).
    • CARTO_STALE_IDS — the stale node ids joined by newlines (staleIds.join("\n")).

  6. Await the child. Using Promise.withResolvers, the function wires the child’s lifecycle to a promise:

    • child.on("error", reject) — a spawn failure rejects.
    • child.on("close", (code, signal) => …) — exit code 0 resolves (settle()); a non-null signal rejects with astro build terminated by signal <signal>; any other non-zero code rejects with astro build exited with code <code>.

    The returned promise therefore settles exactly when the Astro build succeeds and rejects on any failure mode.

child.on("close", (code, signal) => {
  if (code === 0) {
    settle();
  } else if (signal !== null) {
    reject(new Error(`astro build terminated by signal ${signal}`));
  } else {
    reject(new Error(`astro build exited with code ${code ?? "null"}`));
  }
});

Relationships

  • Invoked by the build subcommand of the CLI Commands.
  • Reuses the Hashing Core (diffHashes, staleNodes) and the IR & Source I/O (readIr, currentHashes) to compute staleness.
  • Feeds the Astro Site Template through the three CARTO_* environment variables — the template reads them to load content and mark stale pages.
  • Realizes the reporting side of the Staleness Lifecycle at render time.

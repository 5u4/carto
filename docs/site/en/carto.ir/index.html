Responsibility

src/ir.ts defines the zod schema that is the vault’s validated shape and its single source of structural truth. Every other part of carto — the Hashing Core, the IR & Source I/O layer, and the CLI — trusts an IR only after it has passed IrSchema.parse. The schema encodes not just field types but the cross-node invariants of the Vault Data Model: keys must match ids, references must resolve, anchors must exist, and a source shared by two nodes must agree on its hash.

Key types

The module builds up from the smallest unit to the whole document.

RelativeFile is a refined string used for every source file. It requires a non-empty string and then rejects three shapes of absolute or escaping path:

const RelativeFile = z
  .string()
  .min(1)
  .refine((file) => !file.startsWith("/") && !file.startsWith("\\") && !/^[A-Za-z]:/.test(file), {
    message: "source file must be a relative path",
  })
  .refine((file) => !file.split(/[\\/]/).includes(".."), {
    message: "source file must not contain a '..' path segment",
  });

The first refinement bars a POSIX leading /, a Windows leading \, and a drive-letter prefix like C: (matched by ^[A-Za-z]:). The second splits on either slash and rejects any .. traversal segment. A file is therefore always a clean repo-relative path that an anchor can safely resolve.

SourceSchema is { anchor, file, hash } where anchor and hash are non-empty strings and file is a RelativeFile.

NodeSchema is { id, name, parent, children, sources }. id and name are non-empty strings; parent is a non-empty string or null (a root); children is an array of node-id strings; sources is an array of SourceSchema.

VaultSchema is { name, anchors, generatedAt, locales, defaultLocale }. anchors is a record mapping each anchor name to a non-empty base path (z.record); locales is a non-empty array of locale strings; defaultLocale is a single locale string.

IrSchema wraps everything:

export const IrSchema = z
  .object({
    version: z.literal("1"),
    vault: VaultSchema,
    nodes: z.record(z.string(), NodeSchema),
  })
  .superRefine((ir, ctx) => { /* cross-checks */ });

version must be the literal "1", vault is a VaultSchema, and nodes is a flat record from node id to NodeSchema. The tree is not nested here — it lives in each node’s children, and parent: null marks a root, so the forest may hold many roots.

Data flow — the superRefine cross-checks

Object-level type parsing alone cannot see across nodes, so IrSchema.superRefine runs a second pass that walks the whole document and adds a custom issue for each violated invariant. It executes in order:

  1. defaultLocale membership. If vault.locales does not include vault.defaultLocale, it adds an issue at path ["vault", "defaultLocale"].
  2. It creates an empty Map<string, string> named hashByKey to track the first hash seen for each source, then iterates Object.entries(ir.nodes) as [key, node].
  3. Key/id agreement. If node.id !== key, it flags ["nodes", key, "id"] — the record key and the node’s own id must be the same string.
  4. Parent resolves. If node.parent !== null and Object.hasOwn(ir.nodes, node.parent) is false, it flags ["nodes", key, "parent"]. Using Object.hasOwn rather than the in operator means a parent of "constructor" or "__proto__" — which exists only on the object prototype — is rejected as unknown.
  5. Children resolve. For every string in node.children, if Object.hasOwn(ir.nodes, child) is false it flags ["nodes", key, "children"] — again immune to prototype keys.
  6. Anchor exists. For each source (with its index), if Object.hasOwn(ir.vault.anchors, source.anchor) is false it flags ["nodes", key, "sources", index, "anchor"].
  7. Hash consistency. It computes a per-source key as `${source.anchor}\u0000${source.file}` (the anchor and file joined by a \u0000 NUL, mirroring sourceKey in the Hashing Core). On first sight it records the hash in hashByKey; on a later sight of the same key it compares — an equal hash is fine, a differing hash flags ["nodes", key, "sources", index, "hash"] as a conflict.

Because the identity key joins anchor and file, the same file string cited under two different anchors produces two different keys: it is two distinct sources, is never a hash conflict, and each is validated independently.

Relationships

  • The Vault Data Model is the conceptual counterpart of this schema; IrSchema is its executable form.
  • The Hashing Core reuses the same anchor + \u0000 + file identity key through its sourceKey helper, and operates only on IRs shaped by this schema.
  • The IR & Source I/O layer parses raw JSON through IrSchema.parse on load and serializes validated IRs on save.

Exported types

The module re-exports the inferred TypeScript types so the rest of carto can share one definition:

export type Source = z.infer<typeof SourceSchema>;
export type Node = z.infer<typeof NodeSchema>;
export type Vault = z.infer<typeof VaultSchema>;
export type Ir = z.infer<typeof IrSchema>;

Ir is the shape every consumer imports; there is no hand-written interface that could drift from the runtime schema.

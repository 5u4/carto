Freshness answers "which docs are stale?" Coverage answers the opposite,
prior question: **which code files does no node document at all?** It walks the
code tree, subtracts every file already claimed by a node's `sources`, and
reports the remainder. The gap it finds is not staleness ŌĆö it is absence.

Where [freshness](carto:freshness) compares a tracked file's current hash
against the recorded one to catch drift, coverage never looks at content or
hashes. It only asks set membership: *is this file in the tracked set?* The two
are complementary halves of the same audit ŌĆö drift on the inside, blind spots on
the outside.

## The tracked set

The tracked set is the union of every source across every node in the
[manifest](carto:manifest). `coverageReport` builds it by flattening
`manifest.nodes` and normalizing each `source.file` to a POSIX-relative path
(`coverage.ts:65`). A file is "covered" iff it appears in that set. Everything
the walk finds and the set omits is uncovered (`coverage.ts:66`).

Because the set is exactly the manifest's node sources, coverage is downstream
of documentation authoring: you cover a file by adding it to some node's
`sources`, nothing more.

## The walk and its ignore layers

The other input is the file list, produced by walking the code root. Two pieces
cooperate:

**`buildIgnore`** seeds a baseline and layers project rules on top
(`coverage.ts:33-40`). The baseline `DEFAULT_IGNORES` excludes VCS and
dependency noise plus carto's own artifacts ŌĆö `.git`, `node_modules/`,
`/carto.json`, `/docs/`, `/dist-site/`, `.gitignore`, `.cartoignore`
(`coverage.ts:12`). It then reads `.gitignore` and `.cartoignore` from the root,
if present, and adds their patterns through the [`ignore`](https://www.npmjs.com/package/ignore)
package (`coverage.ts:14`, `coverage.ts:35-38`). `.cartoignore` is the knob for
narrowing the documentable set without touching git behavior.

**`walk`** recurses the code root, testing each entry's POSIX-normalized
relative path against the ignore matcher and skipping matches before it
descends or records (`coverage.ts:42-58`). Normalization via `toPosix` means the
same relative keys are compared on Windows and POSIX alike (`coverage.ts:20-22`,
`coverage.ts:47`).

## Nested doc roots

`coverageReport` takes an optional `docRoot`. The default `/docs/` ignore only
covers a docs directory at the code root. When the doc root sits *nested inside*
the code root at some other path, it computes that relative path and adds it as
an ignore too ŌĆö but only when it is non-empty and does not escape the code root
with `..` (`coverage.ts:62-63`). This stops carto from reporting its own
documentation files as undocumented code. In practice the CLI passes the repo
root as `docRoot` and a possibly-narrower code root beside it
(`packages/cli/src/commands/coverage.ts:18`).

## Why uncovered is a warning, not an error

The result is `{ total, covered, uncovered }` with `uncovered` sorted
(`coverage.ts:66-67`). Uncovered files are a normal, expected state ŌĆö a README,
a test file, an evals harness that genuinely does not belong to any documentation
node. So the [`carto coverage`](carto:cli) command lists them and exits `0` by
default; only the explicit `--fail-on-uncovered` flag turns a non-empty
uncovered list into exit `1` (`packages/cli/src/commands/coverage.ts:7`,
`packages/cli/src/commands/coverage.ts:27`). To silence a file that should never
count, add it to `.cartoignore`; to document it, add it to a node's sources.

:::note
Coverage is content-blind by design. A file can be fully covered yet badly
stale ŌĆö that is [freshness](carto:freshness)'s job to catch. Covered means
*tracked*, not *accurate*.
:::

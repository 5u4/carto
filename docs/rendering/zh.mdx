---
title: 渲染
description: carto 文档根目录如何通过在同一个 core Graph 之上的两阶段渲染，变成一个基于 Starlight-on-Astro 的站点。
---

carto 文档根目录本身并不是一个网站。它是一棵 `<id>/<locale>.mdx` 文件树，外加一份
`carto.json` manifest —— 也就是 [links](carto:links) 与 [freshness](carto:freshness)
子系统所推理的原始素材。`@carto/template` 包的职责，就是把这些原始素材变成一个可浏览的
[Starlight](carto:overview) 站点。它是渲染层：下面的一切都在有人执行 `carto dev` 或
`carto build` 时运行，而这些命令由 [CLI](carto:cli) 启动，并将 `CARTO_ROOT` 指向文档根目录。

## 同一个 Graph，两个阶段

两个阶段都从同一个原语出发 —— `@carto/core` 的 `loadGraph(root)`，它读取根 manifest
以及每一个联邦文档集，汇入单一的 `Graph`
（`{ federated, root, byHash }`，`packages/core/src/graph.ts:15`）。template 层不会重新
解析任何 manifest；它消费这个 `Graph` 及其 `DocSet`
（`packages/core/src/graph.ts:7`）。

这两个阶段是两个独立的程序，依次运行：

1. **`materialize.ts`** —— 一个独立的 Node 脚本，把 `Graph` 投影到 Astro 的内容集合
   `src/content/docs`。它运行至完成并退出，然后 Astro 才启动。
2. **`astro.config.mjs`** —— 在配置求值时再次加载同一个 `Graph`，并从中推导出
   Astro/Starlight 配置（locales、sidebar、redirects、remark 插件）。

之所以要拆分，是因为 Astro 把 `src/content/docs` 目录当作文件输入来拥有，而 carto
的真理之源是 `<docRoot>/docs/<id>/<locale>.mdx`。阶段 1 通过复制并重写来弥合这一
差距；阶段 2 把复制出的树接入一个带主题的站点。

## 阶段 1 —— materialize

`main()` 通过模块末尾的顶层 `await main()` 调用（`packages/template/src/materialize.ts:133`）。
它读取 `CARTO_ROOT`（回退到 `process.cwd()`），加载 graph，然后**清空并重建**内容目录 ——
先 `rm(contentDir, { recursive: true, force: true })` 再 `mkdir`
（`packages/template/src/materialize.ts:28`）。输出是可丢弃的，每次运行都完全重建；不存在
增量合并。

### 空状态

如果根 manifest 没有任何节点，且 graph 不是联邦的，`main` 会写出单个引导页 `index.mdx`
并提前返回（`packages/template/src/materialize.ts:30`，`writeEmptyState` 见
`packages/template/src/materialize.ts:68`）。该页面提示用户让 agent 去记录代码，然后重新
运行 `carto dev`。

### 渲染循环

否则，`main` 会嵌套遍历三个层级 —— **文档集 × 节点 × 站点 locale**
（`packages/template/src/materialize.ts:38`）：

- `graph.byHash.values()` 给出每一个文档集；每个都得到一个 `FederationContext`，携带
  `byHash`、该文档集的 `aliasToHash`，以及站点默认 locale
  （`packages/template/src/materialize.ts:39`）。
- 对每个节点，以及**站点**声明的每个 locale（`graph.root.manifest.locales`），它挑选一个
  `contentLocale`：若该文档集拥有所请求的 locale 则用它，否则回退到文档集自身的
  `defaultLocale` 并发出 `console.warn`（`packages/template/src/materialize.ts:42`）。站点
  可以请求某个联邦文档集从未写过的 `zh` 页面；回退机制保证页面依旧存在。

对每个单元，它读取 `<docRoot>/docs/<id>/<contentLocale>.mdx`
（`packages/template/src/materialize.ts:46`），进行转换，然后写出：

1. **`rewriteLinks`** 就地重写每一个 `carto:` 链接
   （`packages/template/src/materialize.ts:99`）。它调用 `resolveCartoLink` —— 与
   [links](carto:links) 节点所拥有的是同一个解析器 —— 成功时把 `carto:` 目标替换成真实的
   `result.url`。空标签会通过 `titleFor` 从目标标题自动填充
   （`packages/template/src/materialize.ts:110`），后者在标题映射中查找 `hash:id:locale`，
   并依次回退到目标文档集的默认 locale、再回退到裸 id
   （`packages/template/src/materialize.ts:126`）。无法解析的链接原样保留。
2. **`injectStalenessBanner`** 在节点 stale 或 missing 时打上警告横幅（见下文），在 freshness
   映射中以 `${docSet.hash}:${node.id}` 为键（`packages/template/src/materialize.ts:49`）。
3. **`targetPath`** 计算目标位置，随后写入文件
   （`packages/template/src/materialize.ts:52`）。

### 一个页面落在何处

`targetPath` 在 `contentDir` 之下拼接三段
（`packages/template/src/materialize.ts:92`）：

- 一个 **locale 前缀** —— 对站点默认 locale 为空，否则为 `<locale>/`；
- 一个 **文档集前缀** —— 对根为空，否则为去掉前导 `/` 的联邦前缀；
- **根链** —— `rootChain(nodes, id)` 映射为 id 并以 `/` 连接，从而让子节点嵌套在其父节点
  路径之下。

于是，根文档集在站点默认 locale 下的 `overview` 节点写入
`src/content/docs/overview.mdx`，而某个联邦 `web` 文档集的 `auth` 节点的 `zh` 页面写入
`src/content/docs/zh/web/auth.mdx`。

整个 graph 的 freshness 由 `collectFreshness` 一次性预计算，它对每个文档集针对其
`codeRootDir` 运行 `statusReport`（`packages/template/src/materialize.ts:58`）。

## 陈旧横幅

`injectStalenessBanner` 是对单个页面 MDX 的纯字符串变换
（`packages/template/src/staleness-banner.ts:3`）。它的规则是：

- **除非 stale 或 missing 否则不作为。** `fresh`、`unsynced` 或 `undefined` 状态会原样
  返回输入（`packages/template/src/staleness-banner.ts:4`）。`state` 字段来自 `NodeStatus`
  （`packages/core/src/status.ts:15`），归 [freshness](carto:freshness) 所有。
- **感知 frontmatter。** 它匹配前导的 `---\n…\n---` 块；没有 frontmatter 的页面被原样保留
  （`packages/template/src/staleness-banner.ts:5`）。
- **幂等。** 若 frontmatter 主体已存在 `banner:` 键则直接放弃，因此重复运行 materialize
  绝不会重复打戳（`packages/template/src/staleness-banner.ts:8`）。
- **内容。** 它插入一个 Starlight `banner:` frontmatter 键，其消息在“changed since this
  page was written”下列出变更过的源文件（`<code>…</code>`），在“no longer exist”下列出
  缺失的源文件（`packages/template/src/staleness-banner.ts:11`，消息见
  `packages/template/src/staleness-banner.ts:15`）。消息中的引号与反斜杠会被转义，以保证
  YAML 字符串仍然有效（`packages/template/src/staleness-banner.ts:10`）。

Starlight 会把该 `banner:` 键渲染为页面顶部的站点级横幅，于是陈旧页面会在阅读时自行声明
它的陈旧。

## 阶段 2 —— astro.config.mjs

Astro 配置本身就是一个感知 carto 的程序（`packages/template/astro.config.mjs`）。在求值时它
加载同一个 graph、用户配置以及标题映射（`packages/template/astro.config.mjs:9`），然后构建
`defineConfig`：

- **`outDir`** 为 `<root>/dist-site` —— 构建出的站点落在文档根目录旁边，而非 template 包内部
  （`packages/template/astro.config.mjs:15`）。
- **`redirects`** 来自 `buildGraphRedirects`（`packages/template/astro.config.mjs:16`）。
- **`markdown.remarkPlugins`** 为 `[remarkJoinCjkLines]`（`packages/template/astro.config.mjs:18`）。
- **`integrations`** 为 `mermaid`（自动主题图表）加上 `starlight`，其选项为
  `mergeStarlight(user.starlight ?? {}, { locales, sidebar })`
  （`packages/template/astro.config.mjs:22`）。

Starlight 所需的一切都在 `site-config.ts` 中推导。

## 配置推导（`site-config.ts`）

- **`buildLocales`** 把每个 manifest locale 映射为一个 Starlight locale 条目，并把**默认
  locale 键为 `root`**（Starlight 对无前缀 locale 的约定），其余按其自身代码为键
  （`packages/template/src/site-config.ts:7`）。
- **`buildGraphSidebar`** 先构建根文档集的 sidebar，再把其他每个文档集（按 prefix 排序）作为
  自己的带标题分组追加（`packages/template/src/site-config.ts:33`）。在一个文档集内部，
  `entryFor` 会为有子节点的节点生成一个分组，其**第一个 item 是父节点自身**（作为叶子），
  其后跟着它的子节点 —— 因此父节点既是标题又是可点击页面
  （`packages/template/src/site-config.ts:82`）。分组标签来自文档集 home 节点的标题，经由
  `docSetLabel`（`packages/template/src/site-config.ts:44`）。
- **`buildGraphRedirects`** 把 `/`（以及 `/<locale>/`）重定向到根文档集的 home 节点，从而让
  裸站点 URL 落在一个真实页面上（`packages/template/src/site-config.ts:62`，经由
  `buildRedirects`，`packages/template/src/site-config.ts:51`）。
- **`collectGraphTitles`** 读取每个页面 frontmatter 的 `title:`，并以 `hash:id:locale` 为键
  存储（`packages/template/src/site-config.ts:96`，提取见
  `packages/template/src/site-config.ts:114`）。这个映射正是 sidebar 与 materialize 的空标签
  链接填充都会查阅的那份。
- **`loadUserConfig` + `mergeStarlight`** 允许项目放置一个 `carto.config.mjs`，它默认导出
  `{ starlight }`（`packages/template/src/site-config.ts:144`）。合并顺序很关键：carto 先播下
  `title: 'Carto'`，把用户选项覆盖其上，最后**强制占有 `locales` 与 `sidebar`**
  （`packages/template/src/site-config.ts:174`）。用户可以自由定制主题，但无法破坏导航 ——
  carto 从 graph 推导出的这两个结构键始终获胜。

## CJK 换行拼接（`remark-join-cjk.ts`）

Markdown 把单个换行视为软换行，对 CJK 文本而言，这会在字符之间插入一个多余的空格。
`remarkJoinCjkLines` 遍历每个文本节点，**仅当软换行位于两个相邻 CJK 字符之间时**才把它折叠
（`packages/template/src/remark-join-cjk.ts:39`，拼接见
`packages/template/src/remark-join-cjk.ts:25`）。该正则在 CJK 区段外加破折号与省略号上触发，
并要求两侧都是 CJK 字符（`packages/template/src/remark-join-cjk.ts:23`）—— 于是 `你好\n世界`
变成 `你好世界`，而 `hello\nworld` 与 CJK/拉丁混合边界保留其空格。这就是为什么以硬换行源
代码书写的 [zh](carto:overview) 页面依然渲染为连续的正文。

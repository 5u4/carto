title: carto 技能
---

技能（`skills/carto/SKILL.md`）是 carto 的**主接口**——你真正调用的东西。它不是你运行的代码；
它是你的 LLM/agent 遵循的指令集，用来读代码库、写 `.mdx` 页面、编辑那一份 `carto.json`
清单。Carto 的分工是刻意的：**agent 负责判断**（文档什么、如何拆分、什么是真的），而
**CLI 只做必须确定性的事**（对文件做哈希、校验结构与链接）。本页是这套技能的地图；
它所驱动的 CLI 是 [](carto:cli)。

## BYO-LLM 模型

你自带 agent。Carto 在它周围提供三部分：这套技能（写作指南，拆成两份——`skills/carto/SKILL.md`
讲 carto 机制，`skills/documenting-strategy/SKILL.md` 讲它所依托的、与工具无关的内容准则）、
`carto` CLI（护栏）、以及 Astro/Starlight 模板（渲染器）。agent 手写正文与清单；CLI 从不臆造结构——它只做哈希
（`carto sync`）与校验（`carto validate`）。正是这条边界，让一个页面永远无法悄悄地与它所
指向的代码相矛盾：哈希会抓住它。

## 心智模型

你带一个范围调用技能并选定模式。两种模式都汇合到同一个核心：**读代码并运行
`carto status`**，然后**规划节点树**（id、parent、sources）、**为每个节点 × 语言写一个
`.mdx`**、并**编辑 `carto.json`**。接着是 CLI 闸门：**`carto sync`** 填哈希，**`carto
validate`** 校验 schema、树与链接。validate 报错就把你送回它指名的 `.mdx` 或清单；你循环
直到 validate 退出码为 0，然后停止。

- **两种模式。** `document <dir|files>` 是新增覆盖：读代码、构想一棵节点子树、写页面、
  注册 sources。`refresh [<id>]` 在代码变化后重新生成已有页面——`carto status` 显示哪些节点
  非 fresh；不带 id 的 `refresh` 覆盖全部，`refresh <id>` 只针对一个节点及其子树。
- **一个节点是一个心智模型**，一次读完——一个子系统、一条流程、一个核心概念。绝不一个文件
  一个节点。当 `sources` 列表膨胀、或超过约 5 个独立概念堆积时就拆分。
- **受众分层。** 若代码有用户，节点树必须以用户面向层开场（“这是什么、我怎么调用、主循环
  是什么”）；内部架构属于更深的贡献者节点。本站点正是这个形态：[](carto:overview) →
  [](carto:getting-started) 在顶部，[](carto:internals) 在底部。
- **节点类型决定章节顺序。** Explanation（concept → context → mechanism → tradeoffs →
  example）、Tutorial（goal → prerequisites → steps → verification → next）、或
  Reference（overview → signature → parameters → returns → examples → edge cases）。
  多数节点是 Explanation。
- **硬性底线。** 每个节点都带 Intent + 一个心智模型视图（3–5 个概念及其关系，用散文或一个
  简短的带标签列表——mermaid 图是可选的，仅当一个 5–7 节点的小图才值得画）+ 在每个承重论断
  上的 `path:line` 锚点。每个**用户面向**节点还需要一个真实、可复现的完整示例——真实命令配
  真实输出。

## 每个页面必须包含什么

四条原则贯穿所有节点类型：**价值优先**（以读者获得什么开场）、**先概览后细节**、
**具体胜于含糊**（每个论断可观察——真实命令、真实输入/输出，而非“适当地处理了缓存”）、
以及**充分的背景**（在首次出现处定义陌生概念）。正文和链接放进 `.mdx`；结构和源追踪放进
`carto.json`（[](carto:concepts)）——两者永不混杂。

页面通过 Starlight 渲染为 MDX，所以纯 Markdown 并非唯一工具：asides
（`:::note`/`:::caution`）、Expressive Code 的代码块标题与行高亮、`<Steps>`、
以及 `<details>` 都可用——遵循与 mermaid 相同的克制：是调味，不是主菜
（`skills/carto/SKILL.md:215`）。一条硬规则：每个 `carto:` 链接都放在纯 Markdown 链接目标里，
绝不放进 JSX prop，否则校验器和构建期改写器根本看不到它（`skills/carto/SKILL.md:235`）。

## 校验纪律（不可协商）

技能的脊梁是：文档必须**可由代码行为证明**：

- **注释、docstring 和命名是假设，不是证据。** 把它们当线索；对照实际行为核实每个论断；
  绝不逐字照抄。
- **每个论断都带 `path:line` 锚点。** 若不能从代码确认，就别写——标一个 TODO，而不是臆造事实。
- **走查追踪真实代码路径**，输入输出是代码真能产生的——没有理想化的示例。
- **所有语言一起生成。** 先写 `defaultLocale`，再写每种译文；译文**逐字**保留每个 `carto:`
  链接和每个 `path:line` 锚点——翻译正文，绝不翻译标识符。

## 你何时调用它

你调用技能；它产出 `.mdx` + 一次 `carto.json` 编辑；然后你运行 CLI。循环只在 [](carto:cli)
的 `carto validate` 退出码为 0 时结束——那正是技能所要闭合的契约。想在真实源码上端到端地看它，
读 [](carto:getting-started)。

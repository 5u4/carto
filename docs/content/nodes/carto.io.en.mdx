## Responsibility

`src/io.ts` is the **only filesystem boundary** in carto. Every read of `ir.json`, every write back to it, and every read of a cited source file passes through this module. Keeping all disk access here means the rest of the pipeline — the [Hashing Core](/en/carto.core/), the [CLI Commands](/en/carto.cli/), and the [Site Build Driver](/en/carto.build/) — works with plain in-memory data and never touches `node:fs` directly.

## Key functions

- `readIr(irPath)` — reads the file as UTF-8, `JSON.parse`s it, then runs it through `IrSchema.parse` (see [IR Schema](/en/carto.ir/)). Validation is not optional: malformed or structurally invalid IR throws here, so **bad IR fails loudly** at the boundary instead of corrupting a later stage.
- `writeIr(irPath, ir)` — serializes with `JSON.stringify(ir, null, 2)` and appends a trailing newline before writing UTF-8. Pretty two-space indentation plus the terminal newline keeps `ir.json` diff-friendly and stable across rewrites.
- `currentHashes(ir, contentDir)` — resolves and hashes every cited source file, returning a `Map<string, string>` keyed by `sourceKey`. This is the workhorse; the rest of this page walks it.

## Data flow

`currentHashes` computes the *live* hash of each source on disk, in bounded parallelism:

1. **Collect.** `collectSources(ir)` (from the [Hashing Core](/en/carto.core/)) walks every node's `sources` and returns a deduplicated `SourceRef[]` — the same `anchor:file` cited by two nodes is hashed once.
2. **Fan out.** A shared cursor `next` indexes the source list. The module spins up `Math.min(HASH_CONCURRENCY, sources.length)` worker promises, where `HASH_CONCURRENCY = 16`. Each worker loops, claiming the next source via `next++` until the list is exhausted — a bounded pool that never opens more than 16 files at once.
3. **Resolve.** For each source, the worker resolves the anchor directory first — `resolve(contentDir, ir.vault.anchors[source.anchor])` — then resolves `source.file` against that anchor directory. Anchors are therefore interpreted **relative to the content dir**, and the file relative to its anchor.
4. **Read + hash.** The worker reads the bytes with `readFile` and passes them to `hashContent`, storing the result under `sourceKey(source.anchor, source.file)`.
5. **Skip vs. surface.** A read that fails with code `ENOENT` is swallowed with `continue`: a missing file simply produces **no map entry**, so downstream code reads it as absent rather than as an error. Any other error — for example `EISDIR` when a source path points at a directory — is re-thrown and propagates out of the pool.
6. **Join.** `await Promise.all(pool)` waits for every worker, and the populated `Map` is returned.

```ts
const anchorAbs = resolve(contentDir, ir.vault.anchors[source.anchor]!);
const fileAbs = resolve(anchorAbs, source.file);
try {
  hashes.set(sourceKey(source.anchor, source.file), hashContent(await readFile(fileAbs)));
} catch (error) {
  if ((error as NodeJS.ErrnoException).code === "ENOENT") {
    continue;
  }
  throw error;
}
```

## Relationships

- Consumes and validates the shape defined by the [IR Schema](/en/carto.ir/).
- Delegates all hashing and source enumeration to the [Hashing Core](/en/carto.core/) (`collectSources`, `hashContent`, `sourceKey`).
- Is the read/write engine behind every subcommand in the [CLI Commands](/en/carto.cli/) and the staleness computation in the [Site Build Driver](/en/carto.build/).

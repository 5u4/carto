# Staleness Lifecycle

This is a concept node: it traces the freshness lifecycle of a vault end to end, across the CLI and the hashing core, rather than documenting one file.

## Responsibility

Staleness answers one question: *which pages might no longer match the source they describe?* carto tracks this deterministically by hashing every cited source and comparing the current hash against the hash stored in `ir.json`. Crucially, staleness is **advisory** — it reports drift, it never blocks anything, and it never edits your prose.

## What "stale" is measured against

Every source in `ir.json` carries a `hash` in `sha256:<hex>` form. That stored hash is a snapshot of the file taken the last time you ran `carto hash`. A node is considered stale when a source it cites now hashes to something different from the stored value. Staleness is therefore **per-source**: it is a property of individual `anchor:file` pairs, and a node is flagged only because one of its sources drifted.

## Data flow — from `pending` to a build hint

Walking the lifecycle once, in order:

1. **Author writes a source.** When you add a source to a node you set its `hash` to the placeholder `sha256:pending`. You are declaring "this file backs this node" without yet knowing its hash.
2. **`carto hash` records the truth.** The command resolves each source against its anchor, reads the file off disk, computes `sha256:<hex>`, and writes the real values back into `ir.json`. The `pending` placeholders become concrete hashes. See [IR & Source I/O](/en/carto.io/) for the file-reading and hashing step, and [Hashing Core](/en/carto.core/) for the hash function itself.
3. **A source is edited.** Later, someone changes one of those files. Its content now hashes to a new value, but `ir.json` still holds the old one — the two disagree.
4. **`carto diff` finds the drift.** `diff` recomputes current hashes and lists every source whose current hash differs from the stored one (or that is now missing), printed as `anchor:file`. This is the raw file-level signal.
5. **`carto stale` maps drift to pages.** `stale` takes the changed sources and reports the node ids that cite them, each with the offending files. This is a **report, not a gate** — it tells you which pages *might* be out of date; whether and when to rewrite them is entirely your call.
6. **`carto build` surfaces the hint.** During a build, the same staleness check runs and the stale node ids are passed to the site template, which marks those pages with a hint. The page still builds and ships — the hint is a nudge, not an error. See [Site Build Driver](/en/carto.build/).

The `diff`, `stale`, and build checks all lean on the same pair of pure functions in [Hashing Core](/en/carto.core/): one computes the set of changed sources, the other maps changed sources to the nodes that cite them.

## Advisory, never a gate

Nothing in this lifecycle rewrites content or fails a build on your behalf. `stale` is a report you read, and `build` still produces a complete site even when pages are stale. Refreshing a page — rewriting all of its locales together and re-running `carto hash` — is a decision you make, not one the tool forces. Concept nodes such as this one cite no sources, so they never appear in a staleness report at all.

## Relationships

- [Hashing Core](/en/carto.core/) — the `diffHashes` and `staleNodes` functions that compute drift.
- [IR & Source I/O](/en/carto.io/) — reads each source off disk and hashes it.
- [CLI Commands](/en/carto.cli/) — exposes `hash`, `diff`, `stale`, and `build`.
- [Site Build Driver](/en/carto.build/) — runs the check at build time and passes the stale ids to the template.

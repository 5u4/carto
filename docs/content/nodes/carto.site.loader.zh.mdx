# 内容加载器

站点关于一个知识库需要知道的一切，都经由本节点。它把磁盘上的内容目录变成带类型、经校验的数据：覆盖
节点页面的 Astro 内容集合、解析后的 IR、过期节点 id 的集合，以及导航所依据的先序 `tree`。它的两个
源文件是 `astro/src/content.config.ts` 与 `astro/src/lib/carto.ts`。

## 内容在哪里——`contentDir`

`lib/carto.ts` 在模块加载时一次性解析出内容目录：

```ts
export const contentDir = resolve(process.env.CARTO_CONTENT_DIR ?? "example-content");
```

[站点构建驱动](/zh/carto.build/) 会把 `CARTO_CONTENT_DIR` 注入为绝对路径；当它未设置时（开发模板
期间）回退到 `example-content`。本模块中其余所有读取都相对于 `contentDir`。

## 内容集合——`content.config.ts`

`content.config.ts` 定义了唯一一个集合 `nodes`，由 Astro 的 glob 加载器加载：

```ts
const nodes = defineCollection({
  loader: glob({ pattern: "*.mdx", base: `${contentDir}/nodes` }),
  schema: z.object({
    id: z.string().min(1),
    name: z.string().min(1),
    locale: z.string().min(1),
    contentHash: z.string().min(1),
  }),
});
```

- **加载器。** 它在 `<contentDir>/nodes` 下 glob 匹配 `*.mdx`，于是像 `carto.site.en.mdx` 这样的
  每个文件都成为一条条目。扁平的文件名就是全部；没有嵌套目录。
- **Schema。** zod schema 校验每个文件的 frontmatter——`id`、`name`、`locale`、`contentHash`，
  每个都是非空字符串。若 frontmatter 缺少其中之一，或增加结构性键，都会导致构建失败。这正是每个节点
  页面所携带的四键 frontmatter。

## 读取 IR——`loadIr`

`loadIr` 读取并解析描述知识库形态的 IR 文档：

```ts
export function loadIr(): Ir {
  const raw = readFileSync(resolve(contentDir, "ir.json"), "utf8");
  return IrSchema.parse(JSON.parse(raw));
}
```

它从 `contentDir` 读取 `ir.json`，并用 `IrSchema` 解析——与 CLI 使用的是*同一个* schema（从核心的
`src/ir` 导入），因此站点与工具对“什么才是合法的知识库”达成完全一致。IR 所强制的形态见
[IR Schema](/zh/carto.ir/)，这一形态的含义见 [知识库数据模型](/zh/carto.model/)。

## 哪些节点过期——`staleIds`

驱动以换行连接的字符串传入过期 id；加载器把它还原成一个集合：

```ts
export const staleIds = new Set(
  (process.env.CARTO_STALE_IDS ?? "")
    .split("\n")
    .map((value) => value.trim())
    .filter((value) => value.length > 0),
);
```

空输入产生空集合——无一过期。布局会逐节点查询这个集合，以决定是否显示过期提示；见
[过期生命周期](/zh/carto.staleness/)。

## 遍历森林——`tree`

`tree(ir)` 把节点森林摊平成一个有序列表，供导航自上而下渲染。每条目把一个节点与它的 `depth` 配对：

```ts
export function tree(ir: Ir): TreeItem[] {
  const items: TreeItem[] = [];
  const visited = new Set<string>();
  const walk = (node: Node, depth: number): void => {
    if (visited.has(node.id)) {
      return;
    }
    visited.add(node.id);
    items.push({ node, depth });
    for (const childId of node.children) {
      const child = ir.nodes[childId];
      if (child !== undefined) {
        walk(child, depth + 1);
      }
    }
  };
  for (const node of Object.values(ir.nodes)) {
    if (node.parent === null) {
      walk(node, 0);
    }
  }
  return items;
}
```

按顺序读这段遍历：

1. 它从每个根出发——每个 `parent` 为 `null` 的节点——以 `Object.values` 的插入顺序进行，因此森林
   可以有多个根，并按 IR 列出它们的顺序出现。
2. 每次 `walk` 都在当前 `depth`（根为 `0`）记录该节点，然后按数组顺序递归进入 `node.children`，
   并把 `depth` 加一。这是一次**先序**遍历：父节点先于其后代被输出，且整个子树先于下一个兄弟被输出。
3. 缺失的子 id——列在 `children` 中却不存在于 `ir.nodes`——会被跳过而非抛错，因此一个只写了一部分的
   知识库仍能渲染。
4. 一个 `visited` 集合守护着每个节点 id。已经输出过的节点绝不会被再次遍历，这意味着环会终止，而被两个
   父节点共享为子节点的节点只会被放置一次（在第一个到达它的父节点处）。

其结果喂给 [页面与布局](/zh/carto.site.pages/)，后者为每个条目渲染一条导航链接并按 `depth` 缩进。

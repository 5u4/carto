## Responsibility

`src/cli.ts` is carto's **command surface** — the executable entry point that turns terminal invocations into pipeline calls. It is built on [citty](https://github.com/unjs/citty): a root `carto` command with four subcommands. This layer holds no hashing logic of its own; it parses arguments, wires them to the [IR & Source I/O](/en/carto.io/) and [Hashing Core](/en/carto.core/), and prints human-readable output. Critically, **the CLI never calls a model** — it is deterministic reporting only.

## Shared arguments

Every subcommand shares one option, defined once in `sharedArgs`:

- `--content` (string, default `content`) — the content directory that holds `ir.json` and `nodes/`.

Each subcommand resolves the IR path the same way: `resolve(args.content, "ir.json")`. So `ir.json` is always read from **under `--content`**.

## Key subcommands

### `hash`

Recompute source-file hashes into `ir.json`. Flow: `readIr` → `currentHashes` → `applyHashes` → `writeIr`. It reads and validates the IR, computes live hashes for every cited source, folds those hashes back into the node forest with `applyHashes`, writes the result, and prints the count:

```
hashed <n> source file(s)
```

### `diff`

List source files that changed since the last `hash`. Flow: `readIr` → `currentHashes` → `diffHashes`. If there are no changes it prints `no changes`. Otherwise it prints one line per change, tab-separated, tagging each as `missing` (the file no longer hashes — `change.current === null`) or `changed`, followed by the `anchor:file` identity:

```
changed	self:src/core.ts
missing	self:src/gone.ts
```

### `stale`

Map changed files to the node ids they affect — **a report, not a gate**. Flow: `readIr` → `currentHashes` → `diffHashes` → `staleNodes`. It reduces each change to its `{ anchor, file }`, feeds them to `staleNodes`, and prints `no stale nodes` when nothing is affected. Otherwise it prints one line per stale node: the node `id`, a tab, then its hit files joined by `, `. Nothing is written back; the author decides whether to rewrite. See [Staleness Lifecycle](/en/carto.staleness/).

### `build`

Compile `content/` into a static HTML site. It adds one option to the shared set:

- `--out` (string, default `dist-site`) — output directory for the generated site.

It delegates entirely to the [Site Build Driver](/en/carto.build/), calling `build({ irPath, contentDir, outDir })` where `irPath` is `resolve(args.content, "ir.json")`, `contentDir` is `args.content`, and `outDir` is `args.out`.

## Data flow

1. `runMain(main)` boots citty against the root `carto` command, whose `subCommands` are `{ hash, diff, stale, build }`.
2. citty parses `argv`, applies the `--content` (and for `build`, `--out`) defaults, and dispatches to the matching subcommand's `run`.
3. The `run` handler resolves the IR path, calls into [IR & Source I/O](/en/carto.io/) and the [Hashing Core](/en/carto.core/), and writes its report to `process.stdout`.

## Relationships

- Reads/writes IR through the [IR & Source I/O](/en/carto.io/) boundary (`readIr`, `writeIr`, `currentHashes`).
- Applies and interprets hashes via the [Hashing Core](/en/carto.core/) (`applyHashes`, `diffHashes`, `staleNodes`).
- Hands `build` off to the [Site Build Driver](/en/carto.build/).
- Surfaces the [Staleness Lifecycle](/en/carto.staleness/) through `diff` and `stale`.

# Vault Data Model

这是一个概念节点：它描述 carto vault 所构建于其上的数据模型，独立于任何单个源文件。具体的 schema 及其校验见 [IR Schema](/zh/carto.ir/)。

## 职责

一个 carto **vault** 就是一个逻辑上的知识库。这个数据模型只回答一个问题：一个 vault 由哪些部件组成，它们又如何拼合，使得同一套原语既能描述一个仓库、也能描述横跨多个仓库的知识库？agent 写进 `ir.json` 的一切，都是这个模型的实例。

## 组成部件

- **Vault。** 最顶层的容器：一个 `name`、一组 `anchors`、声明的 `locales` 与 `defaultLocale`，以及一个 `generatedAt` 时间戳。它本身不携带内容——只是为其下的一切定框。
- **锚点（Anchors）。** 从短名字到根目录的映射。每个 source 引用一个锚点外加一个仓库相对的 `file`，所以锚点是让 source 得以存在于 vault 自身目录之外的那层间接。锚点路径相对于存放 `ir.json` 的目录来解析，这正是像 `..` 这样的相对锚点可移植的原因。
- **节点（Nodes）。** 扁平记录 `nodes` 按 id 装着每个节点。一个节点有 `id`、显示用的 `name`、一个 `parent`、一份 `children` 列表和一份 `sources` 列表。
- **Sources。** 每个 source 是 `anchor` + `file` + `hash`。同一个 `file` 在两个不同锚点下被引用就是两个不同的 source；被两个节点共享的同一个 source 必须携带完全一致的 hash。
- **语言（Locales）。** 语言在 `vault.locales` 中声明一次；每个节点每种语言有一个 MDX 兄弟文件。

## children 是唯一的深度状态

vault 没有单独的“层级”或“章节”概念。容纳树完全存在于每个节点的 `children` 数组里，而**节点在这棵树中的位置就是唯一的深度状态**。`children` 为空的节点只是还没写它的下一层——没有任何东西刻意把它标成叶子。除容纳之外的关系（依赖、数据流、谁在用它）从不编码进 `ir.json`；它们以行内链接的形式活在正文里，从而让结构保持为一片纯粹的森林。

## 锚点让 vault 摆脱单一仓库

因为每个 source 都通过锚点解析，vault 并不被绑死在单一 checkout 上：

1. **一个仓库** 声明一个锚点——这里是解析到仓库根的 `self`——每个 source 都引用它。
2. **一个知识库** 声明多个锚点，它想描述的每个仓库或 checkout 各一个，来自不同根的节点在森林里并肩而立。
3. **挂接一个依赖** 无非是再加一个锚点，然后就你用到的部分写你自己的节点；你是从你的 vault 出发描述它的内部，而不是导入它的文档。

`carto hash` 如何把一个锚点加文件变成磁盘上真实的哈希，见 [CLI Commands](/zh/carto.cli/)。

## 概念节点是 wiki 的胶水

`sources: []` 的节点——比如本页——描述的是一个想法而非一个文件。因为它不锚定任何 source，概念节点**永远不会陈旧**（见 [Staleness Lifecycle](/zh/carto.staleness/)）。概念节点的存在，是为了让你永远不必把一个想法硬钉到一个它并不居住的文件上；它们正是把一堆按文件划分的页面变成一张互联 wiki 的东西，给你一个可以链接*过去*的落点。

## 语言的原子性

只有当一个节点的*每一种*语言都新鲜时，它才算新鲜。各语言是兄弟关系，而非主/译关系，所以当你重新生成一个节点时，要连同它的全部语言一起重新生成——绝不落下任何一种语言。这条原子性是模型的规则，而非 CLI 能替你修补的东西。

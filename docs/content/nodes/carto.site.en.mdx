# Astro Site Template

The Astro project that carto shells out to when it renders a vault into a static
site. carto itself never imports Astro; instead the [Site Build Driver](/en/carto.build/)
spawns `astro build` against this template directory and hands it the vault
through environment variables. Everything under `astro/` — the config, the content
loader, the routes, and the layout — is a fixed template; the *content* it renders
is whatever content directory the driver points it at.

## Configuration — `astro.config.mjs`

The single source file for this node is `astro/astro.config.mjs`. It is a small
`defineConfig` call with three deliberate choices:

```js
export default defineConfig({
  outDir: process.env.CARTO_OUT_DIR ?? "../dist-site",
  integrations: [mdx()],
  markdown: {
    shikiConfig: { theme: "github-light" },
  },
});
```

- **`outDir` from the environment.** The build output directory is read from
  `CARTO_OUT_DIR`, falling back to `../dist-site` when the variable is unset (for
  example when you run Astro directly during template development). The driver
  always sets it, so a real carto build writes wherever `--out` asked.
- **`@astrojs/mdx`.** The `mdx()` integration is registered so every node page —
  which authors write as `.mdx` — compiles to a component and can carry the
  cross-links and fenced code these pages rely on.
- **Shiki `github-light`.** Markdown code fences are highlighted with the
  `github-light` theme, matching the light colour scheme the layout ships.

## How the driver hands off the vault

carto never passes files as arguments; it passes three environment variables and
lets the template read them. Walking the handoff (see
[Site Build Driver](/en/carto.build/)):

1. `carto build` reads the IR, diffs the recorded hashes against the sources on
   disk, and computes the set of stale node ids.
2. It resolves the template directory (the `astro/` folder shipped with carto) and
   the local Astro binary.
3. It spawns `astro build --root <template>` with the parent environment plus
   three injected variables:
   - `CARTO_CONTENT_DIR` — the absolute path to the content directory being
     rendered.
   - `CARTO_OUT_DIR` — the absolute path where the finished site should land.
   - `CARTO_STALE_IDS` — the newline-joined list of stale node ids.
4. Astro boots with this config, and the two child modules pick the variables up:
   the loader reads `CARTO_CONTENT_DIR` and `CARTO_STALE_IDS`, the pages render
   each node through the layout.

Because the config resolves `outDir` at load time, the value the driver injects is
what governs the output location; the `../dist-site` default only applies when you
run the template outside of carto.

## The two children

This node is the template's root; its two children divide the work:

- [Content Loader](/en/carto.site.loader/) — turns the content directory into
  typed, validated data: the Astro content collection over the node `.mdx` files,
  `loadIr`, the `staleIds` set, and the `tree` walk.
- [Pages & Layout](/en/carto.site.pages/) — the routes and the shell: the index
  redirect, the dynamic node route, and the navigation/locale/stale layout.

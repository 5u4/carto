# Content Loader

Everything the site needs to know about a vault comes through this node. It turns a
content directory on disk into typed, validated data: an Astro content collection
over the node pages, the parsed IR, the set of stale node ids, and the preorder
`tree` the navigation is built from. Its two source files are
`astro/src/content.config.ts` and `astro/src/lib/carto.ts`.

## Where the content lives — `contentDir`

`lib/carto.ts` resolves the content directory once, at module load:

```ts
export const contentDir = resolve(process.env.CARTO_CONTENT_DIR ?? "example-content");
```

The [Site Build Driver](/en/carto.build/) injects `CARTO_CONTENT_DIR` as an
absolute path; when it is unset (template development) it falls back to
`example-content`. Every other read in this module is relative to `contentDir`.

## The content collection — `content.config.ts`

`content.config.ts` defines a single collection, `nodes`, loaded by Astro's glob
loader:

```ts
const nodes = defineCollection({
  loader: glob({ pattern: "*.mdx", base: `${contentDir}/nodes` }),
  schema: z.object({
    id: z.string().min(1),
    name: z.string().min(1),
    locale: z.string().min(1),
    contentHash: z.string().min(1),
  }),
});
```

- **Loader.** It globs `*.mdx` under `<contentDir>/nodes`, so every file such as
  `carto.site.en.mdx` becomes one entry. The flat filename is the whole story;
  there are no nested folders.
- **Schema.** The zod schema validates each file's frontmatter — `id`, `name`,
  `locale`, and `contentHash`, each a non-empty string. Frontmatter that omits one
  of these, or adds structural keys, fails the build. This is exactly the four-key
  frontmatter every node page carries.

## Reading the IR — `loadIr`

`loadIr` reads and parses the IR document that describes the vault's shape:

```ts
export function loadIr(): Ir {
  const raw = readFileSync(resolve(contentDir, "ir.json"), "utf8");
  return IrSchema.parse(JSON.parse(raw));
}
```

It reads `ir.json` from `contentDir` and parses it with `IrSchema` — the *same*
schema the CLI uses (imported from the core `src/ir`), so the site and the tool
agree on exactly what a valid vault is. See [IR Schema](/en/carto.ir/) for the
shape it enforces and [Vault Data Model](/en/carto.model/) for what that shape
means.

## Which nodes are stale — `staleIds`

The driver passes stale ids as a newline-joined string; the loader turns it back
into a set:

```ts
export const staleIds = new Set(
  (process.env.CARTO_STALE_IDS ?? "")
    .split("\n")
    .map((value) => value.trim())
    .filter((value) => value.length > 0),
);
```

Empty input yields an empty set — nothing is stale. The layout consults this set
per node to decide whether to show the staleness hint; see
[Staleness Lifecycle](/en/carto.staleness/).

## Walking the forest — `tree`

`tree(ir)` flattens the node forest into an ordered list the navigation renders top
to bottom. Each entry pairs a node with its `depth`:

```ts
export function tree(ir: Ir): TreeItem[] {
  const items: TreeItem[] = [];
  const visited = new Set<string>();
  const walk = (node: Node, depth: number): void => {
    if (visited.has(node.id)) {
      return;
    }
    visited.add(node.id);
    items.push({ node, depth });
    for (const childId of node.children) {
      const child = ir.nodes[childId];
      if (child !== undefined) {
        walk(child, depth + 1);
      }
    }
  };
  for (const node of Object.values(ir.nodes)) {
    if (node.parent === null) {
      walk(node, 0);
    }
  }
  return items;
}
```

Reading the walk in order:

1. It starts from every root — each node whose `parent` is `null` — in
   `Object.values` insertion order, so the forest can have many roots and they
   appear in the order the IR lists them.
2. Each `walk` records the node at its current `depth` (roots at `0`), then
   recurses into `node.children` in array order, incrementing `depth`. This is a
   **preorder** walk: a parent is emitted before its descendants, and a whole
   subtree is emitted before the next sibling.
3. A missing child id — one listed in `children` but absent from `ir.nodes` — is
   skipped rather than throwing, so a partially-written vault still renders.
4. A `visited` set guards every node id. A node already emitted is never walked
   again, which means a cycle terminates and a node shared as a child of two
   parents is placed once (at the first parent that reaches it).

The result feeds [Pages & Layout](/en/carto.site.pages/), which renders one nav
link per item and indents it by `depth`.

title: 内部实现（贡献者）
---

本节点面向**改动 carto 本身**的人，而非为自己代码写文档的用户——如果你只想驱动 carto，
去读 [](carto:getting-started)。这里讲三个包如何组合：`@carto/core` 是权威库，
`@carto/cli` 是它之上一层薄薄的 citty 接线，而 `@carto/template` 把基于 id 的文档物料化为
Astro + Starlight 能原生路由的布局。

## 心智模型

三个包指向一个方向：一切都依赖 **`@carto/core`**。`@carto/cli` import core 并把它包进 citty
命令；`@carto/template` import core 并用它来 materialize + 配置 Astro 站点。`cli` 和
`template` 不重新实现 core 的任何 schema/hash/URL/resolver 逻辑——它们调用它。运行时唯一的
跨包调用是 `cli` spawn `template` 的 package 脚本（`dev` / `build:site`）；`template` 的
`materialize.js` 随后写出 `src/content/docs/<slug-chain>.mdx`，再由 Astro + Starlight 构建
成站点。

## `@carto/core` —— 库

一个扁平的桶文件（`packages/core/src/index.ts:2`）重导出八个模块：zod `schema`
（现在包含 `federated` 数组）、`hash`（sha256/16）、`manifest`（read/write/
`syncManifest`）、`tree`（`slugOf`、`childrenOf`、`rootChain`、`urlPath`、
`checkTree`）、`resolver`（`parseCartoLink`/`resolveCartoLink`，具备联邦感知能力）、
`graph`（`loadGraph`，把一份清单整个 `federated` 引用图展平成一张按哈希索引的
文档集映射）、`status`（四态分类器）、以及 `coverage`（`coverageReport`，未追踪
文件遍历）。这是其他包所依赖的每条规则的唯一定义；它编码的模型是
[](carto:concepts)。

## `@carto/cli` —— citty 接线

`packages/cli/src/index.ts:11` 构建一个带七个子命令的 `defineCommand` 并交给 `runMain`
（`packages/cli/src/index.ts:24`）。`packages/cli/src/commands/` 下的每个命令文件都很薄：
它从 `process.cwd()` 读 `carto.json`、调用一个 core 函数、打印一行、设置退出码。两个带
CLI 本地逻辑：`validate` 的 `extractCartoTargets`（`packages/cli/src/links.ts:1`）扫描每个
`.mdx` 中的 `carto:` 目标并喂给 core 的解析器，而 `sync` 用 `headCommit` 读 git `HEAD`
来给源文件盖戳（`packages/cli/src/git.ts:3`）。这七个命令的用户面向行为是 [](carto:cli)。

## `@carto/template` —— 先物料化再路由

Starlight 硬编码 `src/content/docs`，并从目录结构推导 locale 与 URL。与其对抗，
模板选择**物料化（materialize）**：`materialize.ts` 的 `main` 读取 `CARTO_ROOT`，
并通过 `loadGraph` 加载整个联邦图——根文档集加上它（传递地）联邦的每一个文档集
（`packages/template/src/materialize.ts:26`、
`packages/template/src/materialize.ts:28`），随后清空并重建 `src/content/docs`
（`packages/template/src/materialize.ts:29`、
`packages/template/src/materialize.ts:30`）。

当根文档集自身没有节点**且**清单未声明任何 `federated` 条目时，它写一张空态
`index.mdx`——以文档根的文件夹名作标题——然后返回
（`packages/template/src/materialize.ts:31`、
`packages/template/src/materialize.ts:33`，函数位于
`packages/template/src/materialize.ts:69`）；否则它遍历图中的每个文档集，通过
`targetPath` 把每个节点的 `.mdx` 写到带该文档集前缀的 **slug 链路径**——
`<docSet.prefix>/<rootChain(...).map(slugOf).join('/')>`，非默认语言带语言前缀
（`packages/template/src/materialize.ts:90`）。

复制的同时，`rewriteLinks` 把每个 `carto:<id>` **以及联邦形式的
`carto:<alias>/<id>`** 目标替换为解析后的 URL——通过携带由图中各文档集构建出的
`FederationContext` 调用 `resolveCartoLink`（`packages/template/src/materialize.ts:97`，
解析调用位于 `packages/template/src/materialize.ts:106`），并用 `titleFor`
（`packages/template/src/materialize.ts:113`）填充空标签；这些标题来自
`collectGraphTitles`，它遍历图中每个文档集并读取每个页面的 frontmatter
（`packages/template/src/site-config.ts:96`，在
`packages/template/src/materialize.ts:37` 被调用）——单一清单版本的
`collectTitles`（`packages/template/src/site-config.ts:90`）仍然按文档集在背后
支撑它，也仍是非联邦内部用途的入口。

新鲜度同样是图感知的：`collectFreshness` 对每个文档集用它自己的 `codeRootDir`
运行它自己的 `statusReport`，以 `<docSet.hash>:<node.id>` 为键
（`packages/template/src/materialize.ts:38`，函数位于
`packages/template/src/materialize.ts:59`）——联邦文档集的过期状态绝不会用错
`codeRoot` 来计算。随后 `injectStalenessBanner` 在节点为 `stale` 或 `missing` 时
给页面 frontmatter 前置一个 `banner.content`，让渲染后的站点显眼地警示哪些源
文件漂移了（`packages/template/src/materialize.ts:50`、
`packages/template/src/staleness-banner.ts:3`）。

Astro 配置加载同一份图，并通过 `buildLocales` / `buildGraphSidebar` 从中构建
Starlight 的 `locales` 与 `sidebar`（`packages/template/src/site-config.ts:7`、
`packages/template/src/site-config.ts:33`）；`buildGraphSidebar` 先用
`buildSidebar` 渲染根文档集的侧边栏，再为每个联邦文档集追加一个由
`docSetLabel` 标注的折叠区块（`packages/template/src/site-config.ts:44`）——
`buildSidebar` 底层仍做逐清单的递归，因此每个条目的标签就是该页面每种语言的
frontmatter `title`——非默认语言随一个 `translations` 映射一同带上——页面没有
标题时回退到 slug（`packages/template/src/site-config.ts:71`），`entryFor` 把
节点树递归成嵌套条目（`packages/template/src/site-config.ts:82`）。
`buildGraphRedirects` 通过 `buildRedirects` 把站点根 `/`（以及每个语言根）送往
根文档集的 `home` 节点，未设 `home` 时送往其第一个根节点
（`packages/template/src/site-config.ts:62`、
`packages/template/src/site-config.ts:51`）。在配置 Starlight 之前，配置先调用
`loadUserConfig` 读取文档根下可选的 `carto.config.mjs`，再由 `mergeStarlight`
把用户的 `starlight` 覆盖铺在 carto 自有的 `locales` 与 `sidebar` 之下——后者
始终胜出（`packages/template/astro.config.mjs:11`、
`packages/template/astro.config.mjs:23`；
`packages/template/src/site-config.ts:144`、
`packages/template/src/site-config.ts:174`）。其结果精确重现 `urlPath` 所计算的
URL，因此内部链接与联邦链接都能解析到真实路由。它还注册一个 remark 插件
`remarkJoinCjkLines`，去掉 Markdown 在 CJK 源码换行处本会插入的多余空格
（`packages/template/astro.config.mjs:18`、
`packages/template/src/remark-join-cjk.ts:39`）。该包的对外表面是一个两行桶
文件，重导出 site-config 辅助函数（`packages/template/src/index.ts:2`）。

## `carto dev`/`build` 如何触达这里

CLI 的 `dev`/`build` 在运行时解析 `@carto/template/package.json`，并以 `CARTO_ROOT` = 文档根
`spawn pnpm run`（`dev` 或 `build:site`）（`packages/cli/src/commands/dev.ts:15`）。正是那个
环境变量把 `materialize.js` 和 Astro 配置指向你的 `carto.json`，而非模板自身的目录。

## 构建顺序的坑

`carto` bin 目标（`packages/cli/dist/index.js`）在全新克隆上并不存在，所以 pnpm 的第一次
`install` 无法链接它。顺序是 `pnpm install` → `pnpm build` → 再次 `pnpm install`（第二次
install 链接现已构建好的 bin）。仓库 `README.md` 记录了这个坑；`pnpm e2e` 是整条流水线仍能
运行的常驻守卫。

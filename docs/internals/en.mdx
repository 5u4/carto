This node is for people **hacking on carto itself**, not for users documenting
their code — if you just want to drive carto, read [](carto:getting-started).
Here is how the three packages fit: `@carto/core` is the library of record,
`@carto/cli` is thin citty wiring over it, and `@carto/template` materializes
id-based docs into the layout Astro + Starlight route natively.

## Mental model

Three packages point in one direction: everything depends on **`@carto/core`**.
`@carto/cli` imports core and wraps it in citty commands; `@carto/template`
imports core and uses it to materialize + configure the Astro site. `cli` and
`template` reimplement none of core's schema/hash/URL/resolver logic — they call
it. At runtime the only cross-package call is `cli` spawning `template`'s package
scripts (`dev` / `build:site`); `template`'s `materialize.js` then writes
`src/content/docs/<slug-chain>.mdx`, which the Astro + Starlight build turns into
the site.

## `@carto/core` — the library

A flat barrel (`packages/core/src/index.ts:2`) re-exporting seven modules: the
zod `schema`, `hash` (sha256/16), `manifest` (read/write/`syncManifest`), `tree`
(`slugOf`, `childrenOf`, `rootChain`, `urlPath`, `checkTree`), `resolver`
(`parseCartoLink`/`resolveCartoLink`), `status` (the four-state classifier), and
`coverage` (`coverageReport`, the untracked-files walk). This is the single
definition of every rule the other packages depend on; the model it encodes is
[](carto:concepts).

## `@carto/cli` — citty wiring

`packages/cli/src/index.ts:11` builds one `defineCommand` with seven subcommands
and hands it to `runMain` (`packages/cli/src/index.ts:24`). Each command file
under `packages/cli/src/commands/` is thin: it reads `carto.json` from
`process.cwd()`, calls a core function, prints a line, and sets an exit code. Two
carry CLI-local logic: `validate`'s `extractCartoTargets`
(`packages/cli/src/links.ts:1`) scans each `.mdx` for `carto:` targets and feeds
them to core's resolver, and `sync` reads the git `HEAD` to stamp sources via
`headCommit` (`packages/cli/src/git.ts:3`). The user-facing behavior of all
seven is [](carto:cli).

## `@carto/template` — materialize then route

Starlight hardcodes `src/content/docs` and derives locale + URL from directory
structure. Rather than fight it, the template **materializes**: `materialize.ts`
reads the doc root's `carto.json` (via `CARTO_ROOT`), wipes and rebuilds
`src/content/docs` (`packages/template/src/materialize.ts:13`). When the manifest
has no root nodes it writes a single empty-state `index.mdx` — titled with the
doc root's folder name — and returns
(`packages/template/src/materialize.ts:15`, `packages/template/src/materialize.ts:34`);
otherwise it writes each node's `.mdx` to a **slug-chain path** —
`rootChain(...).map(slugOf).join('/')`, locale-prefixed for non-default locales
(`packages/template/src/materialize.ts:56`). As it copies, `rewriteLinks`
replaces every `carto:<id>` with the resolved URL and fills empty labels with the
target's title (`packages/template/src/materialize.ts:62`); those titles come from
`collectTitles`, which now lives in `site-config.ts` and reads each page's
frontmatter (`packages/template/src/site-config.ts:62`, called at
`packages/template/src/materialize.ts:20`).
Then `injectStalenessBanner` prepends a `banner.content` to the page's
frontmatter whenever the node is `stale` or `missing`, so the rendered site
visibly warns which source files drifted
(`packages/template/src/materialize.ts:27`, `packages/template/src/staleness-banner.ts:3`).

The Astro config reads the same manifest and builds Starlight's `locales` and
`sidebar` from it via `buildLocales` / `buildSidebar`
(`packages/template/src/site-config.ts:7`, `packages/template/src/site-config.ts:23`);
`buildSidebar` takes the collected titles, so each entry's label is the page's
frontmatter `title` per locale — non-default locales ride along in a
`translations` map — falling back to the slug when a page has none
(`packages/template/src/site-config.ts:43`), and `entryFor` recurses the node
tree into nested entries (`packages/template/src/site-config.ts:54`).
`buildRedirects` sends the site root `/` (and each locale root) to the `home`
node, or the first root node when `home` is unset
(`packages/template/src/site-config.ts:27`). Before configuring Starlight the
config calls `loadUserConfig` to pick up an optional `carto.config.mjs` at the
doc root, then `mergeStarlight` layers the user's `starlight` overrides under
carto's own `locales` and `sidebar`, which always win
(`packages/template/astro.config.mjs:11`, `packages/template/astro.config.mjs:23`;
`packages/template/src/site-config.ts:104`, `packages/template/src/site-config.ts:134`).
The result reproduces exactly the URLs `urlPath` computes, so `carto:` links
resolve to real routes. It also registers one remark plugin, `remarkJoinCjkLines`,
which strips the spurious space Markdown would otherwise insert where a CJK line
wraps in source (`packages/template/astro.config.mjs:18`,
`packages/template/src/remark-join-cjk.ts:39`). The package's public surface is a
two-line barrel re-exporting the site-config helpers
(`packages/template/src/index.ts:2`).

## How `carto dev`/`build` reach here

The CLI's `dev`/`build` resolve `@carto/template/package.json` at runtime and
`spawn pnpm run` (`dev` or `build:site`) with `CARTO_ROOT` = the doc root
(`packages/cli/src/commands/dev.ts:15`). That env var is what points
`materialize.js` and the Astro config at your `carto.json` instead of the
template's own directory.

## Build order gotcha

The `carto` bin target (`packages/cli/dist/index.js`) doesn't exist on a fresh
clone, so pnpm's first `install` can't link it. The sequence is `pnpm install`
→ `pnpm build` → `pnpm install` again (the second install links the now-built
bin). The repo `README.md` documents this trap; `pnpm e2e` is the standing guard
that the whole pipeline still runs.

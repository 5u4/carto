## Why this matters

`skill/SKILL.md` is carto's product brain: it tells the doc-authoring agent what
to write. Right now its node-splitting heuristic is architecture-biased — "top is
orientation, deeper nodes are subsystems, then flows" — and the page checklist's
**hard floor is Intent + Mental model + Code anchors**, with worked examples only
"required" on flow/subsystem pages. The observable consequence: docs generated by
this skill read like an *internal architecture map for contributors* and skip the
**user's** questions — "as a user, how do I actually drive this thing, and show
me a real example." This was seen first-hand on carto's own self-docs (they
documented the three packages' internals but never the user journey or the skill
a user invokes). Fixing the *docs* without fixing the *skill* means the next repo
someone documents with carto has the same gap. This plan fixes the root: the
skill must make a **user-journey / getting-started node** and **worked examples on
user-facing pages** non-negotiable, so every carto doc set — on any repo — earns
a user-first entry, not just an architecture diagram.

**Section A — "## How to split into nodes"** currently reads (verbatim):
```
## How to split into nodes

A node is **one page = one mental model**, readable in one sitting.

- A node = "one thing you'd explain on a whiteboard": a subsystem, a flow, a core
  concept. **Never one file per node** — mirroring the file tree is exactly what
  carto avoids.
- Split a page when its `sources` list balloons or more than about 5 independent
  concepts pile up.
- Shape the tree top-down: the top is orientation ("what is this, how do the
  pieces fit"); deeper nodes are subsystems, then flows. Starting from a root is
  recommended but not required — middle generation is legal (a dangling parent is
  only a warning).
- Keep `sources` precise: register only the files whose behavior the page
  actually describes. Too broad triggers false "stale" churn; too narrow lets
  real changes go undetected. This is the staleness crosshair.
```

**Section B — "## What each page contains (a checklist, not a template)"**
currently reads (verbatim):
```
## What each page contains (a checklist, not a template)

Layer the content; do NOT force a rigid six-section shape — forcing all six
invites filler. Include a layer when the "Required when" column says so:

| Layer | Content | Required when |
|---|---|---|
| Intent | the problem it solves and its role in the system | always — most valuable, hardest to read from code |
| Mental model | 3 to 5 core concepts, their relations, one mermaid diagram | always |
| Run-through | a real input traced through the code to its output | flow and subsystem pages |
| Contract | public interface, inputs and outputs, invariants, error modes | pages with an outward face |
| Gotchas | counterintuitive bits, edge cases, "why is this weird" | when they exist |
| Code anchors | `path:line` on every load-bearing claim | always — jump-to-source and staleness crosshair |

**Hard floor: Intent + Mental model + Code anchors.** Add the rest as the page
warrants.
```

## Steps

### Step 1: Add audience layering + a mandatory user-journey node to "How to split into nodes"

In `skill/SKILL.md`, inside the **"## How to split into nodes"** section, **append
these two bullets** to the end of the existing bullet list (do not alter the
existing four bullets, do not touch the heading or the intro line):

```
- **Audience layering — lead with the user, not the architecture.** If the code
  is a tool, library, product, or anything with users, the tree MUST open with a
  user-facing layer that answers "as a user, how do I actually use this?": what it
  is and who it's for, how you invoke or call it, and the main loop or workflow
  you drive. Internal architecture (how the packages/modules are built) belongs in
  **deeper** nodes aimed at contributors — never as the top of the tree. A pure
  package-dependency diagram is not orientation; it is architecture wearing an
  overview's hat.
- **Every user-facing tree needs a getting-started node.** Add a dedicated node
  (id such as `getting-started` or `usage`) that walks a first-time user from zero
  to a working result: prerequisites, the exact invocation, what happens at each
  step, and one complete run they can reproduce. This node is required whenever the
  documented thing has users; skip it only for purely internal code nobody invokes
  directly.
```

**Verify**: `grep -qF "Audience layering" skill/SKILL.md && grep -qF "getting-started node" skill/SKILL.md && echo ok` → `ok`. Confirm the heading `## How to split into nodes` and the four original bullets are unchanged (`git diff` shows only additions in this section).

### Step 2: Add a Worked-example layer to the page checklist and raise the floor for user pages

In `skill/SKILL.md`, inside **"## What each page contains (a checklist, not a
template)"**:

(a) Add this row to the table, immediately **after** the `Run-through` row and
**before** the `Contract` row:
```
| Worked example | a real, reproducible example a user can copy — real commands with their real output, or real inputs mapped to real outputs | user-facing pages (getting-started, usage, and any CLI/API entry page) |
```

(b) Replace the closing line of the section — currently:
```
**Hard floor: Intent + Mental model + Code anchors.** Add the rest as the page
warrants.
```
with:
```
**Hard floor: Intent + Mental model + Code anchors.** Add the rest as the page
warrants — but on any **user-facing page** (a getting-started/usage page, or a
page documenting a command/API a user calls), a **Worked example is also part of
the floor**: show at least one real, reproducible example with real output, never
an idealized or invented one (this is the same "real code path" discipline the
Run-through and verification rules demand). A user page with no example has not
met the floor.
```

**Verify**: `grep -qF "Worked example" skill/SKILL.md && grep -qF "is also part of" skill/SKILL.md && echo ok` → `ok`. The table still has its original six rows plus this one (seven data rows).

# Implementation Plans — carto

Generated by the improve skill on 2026-07-08, written against commit `442f9de`
(an essentially empty repo — this is a greenfield build, not an audit of existing
code). Each plan is self-contained: an executor with zero prior context can build
its piece from the plan file plus the repo. Read the plan fully before starting,
honor its STOP conditions, and update your row when done.

**What carto is:** a tool that generates *sustainably-evolving* documentation for
one or more codebases — a top-down mental-model map (not API reference) whose
pages carry machine-checkable anchors back to source, so stale pages are
detectable and regenerable. A user invokes the **carto skill** with their own
LLM/agent to write `.mdx` pages and edit one `carto.json` manifest; a small CLI
hashes and validates; an Astro/Starlight template renders the site.

**Stack (fixed):** TypeScript single-stack pnpm monorepo. `@carto/core` (zod
schema + hashing + tree + link resolver), `@carto/cli` (binary `carto`, built
with citty), `@carto/template` (Astro 7 + Starlight). Node ≥ 20, tests with
vitest. Repo rule: **zero code comments** (enforced by `scripts/lint-comments.sh`).

## Recommended sequencing

```
001 ──▶ 002 ──▶ 003 ──┐
              └▶ 004 ──┼──▶ 006
              └▶ 005 ──┘
```

1. **001** first — nothing compiles or tests without the workspace skeleton and
   the canonical scripts (`build`/`typecheck`/`test`/`lint`).
2. **002** next — the shared library. It is the single definition of the schema,
   the URL algorithm, and the `carto:` link resolver; 003, 004, 005 all bind to
   its contract.
3. After 002, **003 + 004 + 005 run in parallel** — they depend only on 002's
   API, not on each other. (004 is the highest-risk plan; start its routing spike
   early.)
4. **006** last — it needs the CLI (003), the template (004), and the skill (005)
   to exist, and exercises the whole pipeline end to end.

## Dependency notes

- **002 → 003, 004:** the CLI and template import `@carto/core` and reimplement
  none of its schema/hash/URL/resolver logic. If 002 does not export the API each
  consumes (listed in their "Current state" sections), those plans STOP rather
  than duplicate logic.
- **002 → 005:** the skill documents 002's schema and the CLI contract; it can be
  *written* without the code existing, but its facts must match what 002/003 ship.
- **003, 004 → 006:** e2e runs `carto sync → status → validate → build` over
  carto's own docs; it needs the real binaries.
- **Cross-plan contracts locked during authoring (do not silently change one side):**
  - **ESM `.js` imports:** every intra-package relative import carries a `.js`
    extension (`moduleResolution: Bundler` keeps typecheck green; Node ESM needs
    the extension at runtime). Applies to 002/003/004.
  - **`CARTO_ROOT` + template scripts:** the CLI (003) launches the template (004)
    with `CARTO_ROOT` = the doc root, invoking the template's `dev` and
    **`build:site`** package scripts (the template's plain `build` stays `tsc`).
  - **Site output dir:** `carto build` renders into `$CARTO_ROOT/dist-site/`
    (gitignored) — Astro's `outDir` is set to `join(CARTO_ROOT, 'dist-site')` in
    004, which 006 relies on.
  - **mdx layout:** `docs/<id>/<locale>.mdx` (id-based files); URLs come from the
    carto.json slug chain via `@carto/core.urlPath`, never from file paths.

## Design decisions baked into these plans (context for reviewers)

- **Terminology is minimal:** `carto.json` (one per repo = one site) and `node`
  (one page, immutable `id`, tree via `parent`). No `project` layer. Grouping is
  the node tree.
- **Node identity vs. position:** `id` is immutable and is the `carto:` link
  target; `slug` and `parent` describe position and change freely during refactors
  without breaking links. This is what makes doc refactoring cheap.
- **Staleness is hash-based, not git-based:** each source stores a 16-char sha256;
  `carto status` classifies nodes unsynced/stale/missing/fresh. Known accepted
  limitation: a newly added file inside a component is invisible until that
  component is explicitly regenerated.
- **Generation is per-component, not whole-repo:** a "component" is a node plus
  its subtree; you can generate from the middle of the tree (a dangling parent is
  a warning, not an error).
- **Federation is v2:** cross-`carto.json` references (`carto:<alias>/<id>`, the
  `/` form) are reserved in the syntax but rejected by `validate` in MVP.
- **The routing risk (plan 004):** Starlight hardcodes `src/content/docs` and
  derives locale from directory structure. Rather than fight it, 004 *materializes*
  the id-based mdx into Starlight's expected slug-chain layout, so native routing
  reproduces exactly the URLs `@carto/core.urlPath` computes. Plan 004 gates this
  behind a routing spike with a hard STOP if the assumption fails.

## Not covered (explicitly deferred to v2, not planned here)

- **Federation:** registering external `carto.json`s and the `refs` table +
  `carto:<alias>/<id>` resolution.
- **Permalinks:** per-repo `repo:{url,commit}` to render `path:line` code anchors
  as source links (MVP renders them as plain text).
- **Translation-drift detection:** hand-editing one locale's prose out of sync
  with another is not detected (only code→docs staleness is).
- **Localized sidebar labels & backlinks footer:** deferrable enhancements inside
  plan 004 (its Step 8), attempted only after routing is proven.
- **`dev` live re-materialization:** `carto dev` does not watch `carto.json`/docs
  for structural changes mid-session; re-run to pick them up.
